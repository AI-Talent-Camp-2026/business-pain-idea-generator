# User Story Map: Генератор болей → бизнес-идеи

## Backbone (Путь пользователя)

Открыть приложение → Быстро запустить «прогон» → Получить 10–20 идей → Выбрать 1–2 → Изучить аналоги и план → (опционально) Экспорт/копировать → Повторить с новым фокусом

---

## Release 0.1 (Day‑1 MVP — “первая рабочая версия за 1 день”)

Фокус на **работоспособности core flow** и простоте. Всё лишнее — позже.
Критично: **без регистрации/логина/аккаунтов** в Day‑1, иначе это не “за 1 день”.

### [EPIC-000] Прогон без обязательных настроек
---
**[US-000][Фаундер] Как фаундер, я хочу получить 10–20 идей без необходимости выбирать направление, чтобы начать сразу и не думать “а что вводить”.**
- [ ] Кнопка запуска доступна без заполнения любых полей.
- [ ] Если фокус не задан — система выбирает “рандомно” и показывает его в результатах (как объяснение, а не вопрос).
---
**[US-000A][Фаундер] Как фаундер, я хочу, чтобы вывод не перегружал меня доказательствами, чтобы я мог быстро читать идеи.**
- [ ] В карточке идеи по умолчанию видны только паттерны/сигналы в 2–5 строк.
- [ ] Ссылки/примеры доступны по действию “Подробнее”.
---
**[US-000B][Фаундер] Как фаундер, я хочу, чтобы “план+сроки” был максимально простым “скелетом”, чтобы я понял выполнимость за секунды.**
- [ ] План читается за 30–60 секунд (без деталей).
- [ ] Есть минимум 2 варианта: 7 дней и 30 дней.

## Release 1.0 (MVP)

### [EPIC-001] Быстрый старт без вопросов
---
**[US-001][Фаундер] Как фаундер, я хочу запустить генерацию идей “в один клик”, чтобы получить первый результат без опросника.**
- [ ] На старте достаточно минимального ввода (или вообще без ввода).
- [ ] Система не блокирует запуск из‑за «не заполнено поле».
- [ ] Показано ожидание/статус (что “идёт анализ”).
- Notes: ключевой принцип продукта — «сервис думает сам».
---
**[US-002][Фаундер] Как фаундер, я могу (опционально) указать направление/интерес, чтобы идеи были ближе к моему контексту.**
- [ ] Поле «интерес/направление» необязательное.
- [ ] Можно выбрать из предложенных системой направлений (без обязательного выбора).
- [ ] Система объясняет, как направление повлияло на результат (кратко).
---
**[SS-001][Система] Как система, я хочу автоматически выбирать источники и сигналы для анализа, чтобы пользователь получал результат без настройки.**
- [ ] Указано, какие типы источников учитывались (категориями, без тех. деталей).
- [ ] Есть режим «случайный микс» для разнообразия.
- [ ] Источники могут быть RU+EN (микс), но итоговая формулировка болей/идей всегда на русском.

### [EPIC-002] Результат: боли → идеи (10–20)
---
**[US-003][Фаундер] Как фаундер, я хочу получить 10–20 идей за один прогон, чтобы быстро сравнить варианты и выбрать направление.**
- [ ] Количество идей в пределах 10–20.
- [ ] Каждая идея имеет единый шаблон (боль → кому → почему сейчас → идея → аналоги → план/сроки).
- [ ] Каждая идея имеет уровень уверенности (высокий/средний/низкий).
---
**[US-004][Фаундер] Как фаундер, я хочу видеть конкретную формулировку боли, чтобы понимать, что именно “не работает” у людей.**
- [ ] Боль сформулирована одним абзацем без общих слов.
- [ ] Указано, для кого боль (сегмент/персона).
---
**[US-005][Фаундер] Как фаундер, я хочу видеть “сигналы подтверждения” по боли, чтобы доверять, что это не выдумка.**
- [ ] Сигналы описаны понятным языком: типичные фразы/паттерны поведения.
- [ ] По умолчанию показывается краткое описание паттернов (без перегруза).
- [ ] По кнопке “Подробнее” доступны ссылки на источники и примеры (цитаты/обсуждения), если это допустимо.
---
**[SS-002][Система] Как система, я хочу структурировать вывод по единому шаблону, чтобы идеи были сопоставимы.**
- [ ] Шаблон одинаковый для всех идей.
- [ ] В одном месте видны ключевые поля (короткая карточка) и разворот (детали).

### [EPIC-003] Аналоги и «план + сроки» (самое важное)
---
**[US-006][Фаундер] Как фаундер, я хочу видеть 2–3 аналога (конкурента/заменителя) по каждой идее, чтобы быстрее оценивать рынок и не повторять существующее.**
- [ ] В каждой идее есть минимум 2 аналога.
- [ ] Для каждого аналога — ссылка и краткое “что делают”.
---
**[US-007][Фаундер] Как фаундер, я хочу получить план реализации с ориентировочными сроками, чтобы понять, что реально сделать за неделю/месяц в формате one-person company.**
- [ ] План дан в самом простом формате (“скелет” шагов) и легко считывается за 30–60 секунд.
- [ ] Для плана есть минимум 2 варианта: 7 дней (очень быстрый) и 30 дней (реалистичный).
- [ ] Детальная проработка планов (ресурсы/риски/метрики) — позже, не блокирует MVP.

### [EPIC-004] Выбор и повторный прогон
---
**[US-008][Фаундер] Как фаундер, я хочу фильтровать и сортировать идеи (по уверенности/сегменту/сложности), чтобы быстрее выбрать 1–2 кандидата.**
- [ ] Фильтры базовые: уверенность, тип клиента (B2B/B2C), “время до MVP”.
- [ ] Сортировка по одному критерию за раз.
---
**[US-009][Фаундер] Как фаундер, я хочу перезапускать генерацию с новым фокусом, чтобы получить другой набор идей и расширить поле вариантов.**
- [ ] Кнопка “Сгенерировать ещё”.
- [ ] Система старается избегать дублей (явно помечает похожие).
---
**[SS-003][Система] Как система, я хочу сохранять историю прогонов, чтобы пользователь мог вернуться к результатам.**
- [ ] Видна история прогонов с датой/фокусом.
- [ ] Можно открыть прошлый прогон и посмотреть идеи.

### [EPIC-005] Экспорт и шаринг
---
**[US-010][Фаундер] Как фаундер, я хочу экспортировать выбранные идеи в документ, чтобы обсуждать их с командой/менторами и работать дальше.**
- [ ] Экспорт включает: боли, аналоги, план/сроки.
- [ ] Экспорт на русском, в читаемом виде.
- Notes: формат экспорта уточняется (док/таблица/ссылка).

## Gap Analysis & Risks

### Пропущенные функции
- Более глубокая проверка спроса (например, подсказки, где искать первых клиентов) — пока как часть “плана”.
- Режим “для бизнеса” (когда есть данные компании: отзывы, саппорт, CRM) — вне MVP, но важно для масштабирования.

### Открытые вопросы
- Нужно ли различать режимы: “для фаундера” vs “для компании с данными”?
- Какая степень доказательности по болям обязательна на MVP (краткие паттерны по умолчанию + “Подробнее” со ссылками/цитатами)?

### Риски
- [⚠️ Risk] Пользователь не доверяет выводам без прозрачных «почему так решили» и ссылок.
- [⚠️ Risk] Аналоги будут неточными/устаревшими без регулярного обновления.
- [⚠️ Risk] План/сроки слишком “общие” → не помогают принять решение.
